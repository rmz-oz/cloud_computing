Topic 4: Dockerfile & Image Building

1 · What is a Dockerfile?

A Dockerfile is a text file with instructions that Docker (BuildKit) reads to build a Docker image. Each instruction creates a layer in the image.

# A sample Dockerfile for a Python app
FROM python:3.12.2-slim      # base image
WORKDIR /app                  # set working directory
COPY requirements.txt .       # copy requirements first
RUN pip install -r requirements.txt  # install dependencies
COPY . .                      # copy application code last
CMD ["python", "app.py"]      # default command to run
The order of Dockerfile instructions matters for layer caching. Put frequently changing files (source code) at the END. Put stable files (requirements) BEFORE the code.

3 · Layer Caching

Docker caches each step (instruction) in the Dockerfile to speed up subsequent builds. This is called the build cache:

  • If a layer hasn't changed, Docker reuses the cached version — very fast
  • When a layer CHANGES, that layer and ALL subsequent layers are invalidated and rebuilt
  • This is why order matters: put stable steps first, changing steps last
Bad order (inefficient):
COPY . . → RUN pip install -r requirements.txt
Every code change triggers a full pip install!

Good order (cache-friendly):
COPY requirements.txt . → RUN pip install -r requirements.txt → COPY . .
Only requirements changes trigger pip install. Code changes only rebuild the last COPY layer.

4 · COPY vs ADD

Both COPY and ADD copy files into the image, but they have differences:

FeatureCOPYADD
Copy local filesYesYes
Download from URLNoYes
Extract tar.gz archivesNoYes (auto-extracts)
Recommended forMost use casesOnly when URL/archive needed
Best Practice: Prefer COPY over ADD for copying local files. ADD's extra features (URL download, auto-extract) can cause unexpected behavior. Use COPY for clarity and predictability.

5 · CMD vs ENTRYPOINT

# CMD — default command, can be fully overridden
CMD ["python", "app.py"]
# docker run myimage python other.py   ← overrides CMD

# ENTRYPOINT — main executable, harder to override
ENTRYPOINT ["gunicorn"]
CMD ["app:app", "-w", "4"]             # default args to ENTRYPOINT
# docker run myimage app:api           ← replaces CMD args only

# Array (exec) form — preferred
CMD ["gunicorn", "-w", "4", "app:app"]

# String (shell) form — avoid
CMD "gunicorn -w 4 app:app"
Best Practice: Prefer Array (exec) form over String (shell) form for CMD and ENTRYPOINT. Array form: ["command", "arg1"] — runs directly. String form: "command arg1" — runs via /bin/sh -c, which can cause signal handling issues.

6 · Base Images & Best Practices

Choose Smaller Base Images

TagSizeUse When
python:3.12.2-bookworm1.02 GBNeed full Debian environment
python:3.12.2-slim130 MBMost production use cases ✓
python:3.12.2-alpine51.8 MBSmallest; use if no C extensions

Minimize Number of Layers

RUN, COPY, and ADD each create a new layer that increases image size. Combine RUN commands with &&:

# Bad — creates 2 layers
RUN apt-get update
RUN apt-get install -y netcat

# Good — creates 1 layer
RUN apt-get update && apt-get install -y netcat

Use Unprivileged Containers

By default Docker runs as root — a security risk. Use a non-root user:

RUN addgroup --system app && adduser --system --group app
USER app

One Process Per Container

Run each service in its own container. Easier to scale and reuse individual services.

7 · Building Images

docker build -t myapp:latest .         # build from current directory
docker build -t myapp:v1.0 .           # build with version tag
docker build -f MyDockerfile -t myapp . # use specific Dockerfile
docker history myapp:latest            # show layers and sizes
