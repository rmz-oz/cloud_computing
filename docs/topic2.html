Topic 2: Containers & Docker Architecture

1 · What is a Container?

A container is a standard unit of software that packages up code and all its dependencies. As a result, the application runs quickly and reliably in any computing environment.

Containers solve the famous developer problem: "it works on my laptop!" — by ensuring the application runs identically in development, staging, and production.

2 · Docker Container Image

A Docker container image is a lightweight, standalone, executable package that includes everything needed to run an application:

  • Code — e.g., Python application code
  • Runtime — e.g., Python interpreter/environment
  • System tools — OS utilities needed
  • System libraries — shared libraries
  • Settings — configuration and environment variables
Image vs Container: An image is a static blueprint (like a class). A container is a running instance of that image (like an object). Container images become containers at runtime when run on Docker Engine.

4 · Linux Namespaces

Docker uses Linux namespaces to provide isolation for containers. With namespaces, each container:

  • Has its own process tree — can't see processes in other containers
  • Has its own IP, interfaces, and routing tables
  • Cannot access another container's shared memory
  • Has its own / filesystem, separate from the host
Namespaces = "Limits what you can see." Each container lives in its own isolated bubble — it sees only its own processes, its own network, its own filesystem.

5 · Control Groups (cgroups)

Docker uses the Linux kernel feature called cgroups (Control Groups) to limit and allocate resources to containers:

ResourceDocker FlagExample
CPU Limit--cpusdocker run --cpus="1.5" my_container
Memory Limit--memorydocker run --memory="512m" my_container
I/O Limit--device-read-bpsdocker run --device-read-bps /dev/sda:10mb ...
cgroups = "Limits how much you can use." While namespaces control visibility, cgroups control resource consumption — CPU, memory, disk I/O.

How cgroups Work

  • Resource Isolation: Prevents a container from using excessive system resources
  • Resource Allocation: Controls how much CPU, memory, or disk I/O a container can use
  • Process Management: Ensures processes within a container do not exceed assigned limits

6 · Containers vs Virtual Machines

Containers (Docker)
  • Share the host OS kernel
  • Lightweight — MB in size
  • Start in seconds
  • No separate OS per app
  • Virtualize at the application layer
Virtual Machines (VMs)
  • Each VM has its own full OS
  • Heavy — GB in size
  • Start in minutes
  • Requires hypervisor (VMware, VirtualBox)
  • Virtualize at the hardware level

Docker Benefits

  • Portable: Run anywhere — Linux, Windows, cloud, datacenter
  • Lightweight: Share the host OS kernel; no separate OS per application (reducing server and licensing costs)
  • Secure: Strongest default isolation capabilities in the industry

7 · containerd Daemon

containerd is available as a daemon for Linux and Windows. It manages the container lifecycle on the host system:

  • Image transfer and storage
  • Container execution and supervision
  • Low-level storage management
  • Network attachments

containerd implements the Kubernetes Container Runtime Interface (CRI) and is widely adopted across public clouds (AWS, Google Cloud, Azure).
