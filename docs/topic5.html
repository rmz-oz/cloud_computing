Topic 5: Docker Compose, Networking & Volumes

1 · Docker Compose

Docker Compose is a tool for defining and running multi-container applications using a YAML file (docker-compose.yml or compose.yml). With a single command, you start all services.

version: '3.8'
services:
  web:
    image: nginx:latest
    ports:
      - "8080:80"
    depends_on:
      - app

  app:
    build: .
    environment:
      - DB_HOST=db
      - DB_PORT=5432
    depends_on:
      - db

  db:
    image: postgres:15
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
Key commands:
docker compose up — start all services (foreground)
docker compose up -d — start in detached (background) mode
docker compose down — stop and remove containers and networks
docker compose down -v — also remove volumes
docker compose logs -f — follow logs from all services
docker compose ps — list running services

2 · docker-compose.yml Structure

KeyPurpose
servicesDefines each container (service) in the application
imageDocker image to use for this service
buildBuild from Dockerfile in specified directory (instead of image)
portsPort mapping: "hostPort:containerPort"
environmentEnvironment variables for the service
volumesVolume mounts for persistence
depends_onStart this service only after listed services are running
networksAttach service to specific Docker networks
restartRestart policy: no, always, on-failure, unless-stopped

3 · Docker Networking

Docker creates an isolated virtual network for each application. Containers on the same network can communicate using container names as hostnames (Docker's built-in DNS).

# Containers on the same network communicate by name:
# If app service wants to connect to db service:
DB_HOST=db     # just use the service name!
DB_PORT=5432   # container's internal port (not host port)

# Docker Compose automatically creates a shared network
# All services in the same compose file can reach each other by name

Network Types

DriverUse Case
bridge (default)Single-host container communication. Containers on same bridge can communicate by IP or name (with Compose)
hostContainer shares host network — no isolation. Highest performance.
noneNo networking — fully isolated container
overlayMulti-host networking — used with Docker Swarm/Kubernetes
Compose DNS: Docker Compose automatically sets up DNS resolution. A service named 'db' is reachable at hostname 'db' from other services. No IP addresses needed — use service names as hostnames.

4 · Docker Volumes

Container data is ephemeral — lost when the container is removed. Volumes provide persistent storage that survives container lifecycle.

Types of Volumes

TypeSyntaxDescription
Named Volumedb-data:/var/lib/postgresql/dataDocker manages the storage location. Survives container removal. Best for production data.
Host Volume (Bind Mount)/host/path:/container/pathDirectly mounts a host directory. Changes on host appear in container immediately. Best for dev.
Anonymous Volume/var/lib/postgresql/dataDocker generates a random name. Hard to reference. Not recommended for important data.
# Named volume (in docker-compose.yml)
volumes:
  db-data:/var/lib/postgresql/data

# Host volume (bind mount) — for development
volumes:
  - ./app:/app      # current directory ./app → /app in container
  - ./nginx.conf:/etc/nginx/nginx.conf

# Anonymous volume — Docker manages
volumes:
  - /var/lib/postgresql/data
Named volumes vs bind mounts: Named volumes are managed by Docker (stored in /var/lib/docker/volumes/) — portable, easy backup. Bind mounts link to a specific host path — great for development (live code changes).

5 · depends_on & Service Ordering

services:
  app:
    build: .
    depends_on:
      - db      # app starts after db is running

  db:
    image: postgres:15
Note: depends_on only waits for the container to START — not for the service inside to be READY. Database may still be initializing when app tries to connect. Use health checks or retry logic in your app.

6 · Environment Variables in Compose

services:
  app:
    image: myapp
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
      - SECRET_KEY=mysecretkey
    env_file:
      - .env     # load from a .env file (good practice)
Never commit .env files with secrets to git. Use .env.example (without secrets) and add .env to .gitignore.
